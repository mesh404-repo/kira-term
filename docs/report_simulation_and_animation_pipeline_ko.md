# TBS 시뮬레이션 & 애니메이션 연계 보고서

이 문서는 TBS 장비 시뮬레이션이 **어떻게 동작하고**, 시뮬레이션에서 발생한 이벤트가 **어떻게 3D 애니메이션(JSON 시퀀스)으로 연결**되며, 그 결과가 **로그/엑셀**로 남는지 이해하기 쉬운 방식으로 설명합니다.

> 캡처/영상은 KitApp 실행 후 사용자가 직접 추가할 수 있도록, 문서 말미에 “권장 캡처/영상 체크리스트”를 포함합니다.

---

## 1. 한 문장 요약 (왜 필요한가)

**설비 공정 흐름을 ‘시간 축으로 재현’하면서, 동일 이벤트를 ‘로그 + 3D 동작’으로 일관되게 검증/시연하고, 결과를 엑셀로 남기는 시뮬레이션/시각화 도구**입니다.

---

## 2. 구성 요소(누가 무엇을 하나)

### 2.1 시뮬레이션 엔진 (공정 판단/시간 진행)
- 역할: 현재 포트 상태, 대기열, 회수 티켓 등을 기준으로 “다음에 어떤 공정을 수행할지”를 결정합니다. 또한 공정 시간을 시뮬레이션 시간으로 소모하고(예: 이동에 8초가 걸리면 8초 대기), 단계가 시작/완료될 때 이벤트를 발생시킵니다.
- 핵심 개념(설명):
  - **시간 기반 진행**: 각 공정은 즉시 완료되는 것이 아니라, 설정된 시간만큼 진행된 뒤 다음 단계로 넘어갑니다.
  - **이벤트 발생**: “지금 어떤 일이 시작/완료되었는지”를 외부(UI/애니메이션)로 전달하는 신호를 생성합니다.
- 코드 위치: `source/extensions/morph.tbs_control_1/morph/tbs_control_1/simulation_engine.py`

### 2.2 이벤트 표준(XML)
- 역할: 이벤트 정보를 표준화된 XML 형태로 구성한 뒤, 이를 다시 역파싱하여 “정식 시퀀스명/포트 정보(from/to/port)”를 일관된 규칙으로 확보합니다.
- 목적(설명): 이벤트가 여러 위치에서 다양한 형태로 표현되더라도, 최종적으로는 같은 기준으로 애니메이션 매핑/로그를 처리할 수 있도록 입력을 정규화합니다.
- 코드 위치: `source/extensions/morph.tbs_control_1/morph/tbs_control_1/xml_generator.py`

### 2.3 애니메이션 실행 파이프라인 (이벤트 → JSON 선택 → 실행)
- 역할: 시뮬레이션 이벤트 payload를 받아 “어떤 애니메이션 시퀀스(JSON)를 실행할지”를 선택하고, 실제 실행 엔진에 전달합니다.
- 구성:
  - **상태 기반 규칙 우선**: 현재 이벤트 종류, 포트(from/to/port), 포트 점유 상태(어떤 포트가 차 있는지/비었는지)에 따라 다른 JSON을 선택할 수 있습니다.
  - **단순 매핑 fallback**: 복잡한 조건이 필요 없을 때는 이벤트명만으로 JSON을 매핑할 수 있습니다.
  - **실행**: 선택된 JSON(스텝 리스트)을 `SequenceRunner`에 전달하여 MOVE/ROTATE/DELAY 등을 적용합니다.
- 코드 위치: `source/extensions/morph.tbs_control_1/morph/tbs_control_1/control_window.py`

### 2.4 시퀀스 편집기 (애니메이션 JSON 생성/편집)
- 역할: MOVE/ROTATE/DELAY/USD_TIMELINE 같은 스텝을 조합해 “애니메이션 동작을 정의하는 JSON”을 만들고 저장합니다.
- 특징:
  - “현재 위치부터 시작” 옵션: 뷰포트에서 사용자가 만든 자세(변환)를 스냅샷으로 저장해, 다른 시점/환경에서도 같은 시작 자세로 재생되도록 합니다.
- 코드 위치:
  - 편집 UI: `source/extensions/morph.tbs_control_1/morph/tbs_control_1/sequence_editor.py`
  - 실행 엔진: `source/extensions/morph.tbs_control_1/morph/tbs_control_1/sequence_engine.py`

## 3. 전체 동작 흐름

### 3.1 공정 흐름(예시)
아래는 “직렬 모드” 기준으로 대표적인 흐름 예시입니다.

1) **투입**: OHT가 LOT을 BP1 또는 EP로 투입
2) **이송**: BP1 → 버퍼(BP2~BP4)
3) **공정 투입**: 버퍼(BP) → EP
4) **회수**: EP → OHT 회수

이 각 단계에서 시뮬레이션 엔진은 이벤트를 발생시키고, UI는 해당 이벤트에 연결된 애니메이션을 실행할 수 있습니다.

### 3.2 이벤트가 애니메이션으로 연결되는 과정
1) 시뮬레이션이 이벤트 payload를 생성합니다. payload에는 보통 이벤트 종류(`seq`)와 포트 정보(`from_port_id`, `to_port_id`, `port_id`), LOT 식별자(`lot_id`), 그리고 포트 점유 스냅샷(`ports_occupancy`)이 포함됩니다.  
2) UI는 payload를 수신한 뒤 XML을 생성/역파싱하여 이벤트 정보를 표준화합니다(동일 이벤트를 동일 규칙으로 다루기 위함).  
3) 규칙 파일에서 “현재 이벤트 + 현재 포트 상태”에 맞는 JSON을 선택합니다.  
4) 선택된 JSON을 실행합니다. JSON은 스텝 리스트이고, 실행 엔진이 MOVE/ROTATE/DELAY 등을 시간 축으로 적용합니다.  
5) 실행 결과는 UI 패널과 엑셀 산출물에 기록됩니다(추적/보고 목적).

---

## 4. 로그/증적(Traceability) 구조

### 4.1 UI 로그 3종
1) **진행현황**: 현재 단계와 진행률(%)을 시간 간격으로 갱신합니다.  
2) **이력로그**: 공정 단계의 시작/대기/완료 등 상황 변화가 기록됩니다.  
3) **애니메이션 실행이력**: 이벤트별로 어떤 JSON이 선택/실행되었는지, 파일 존재 여부, 실행 결과가 기록됩니다.

### 4.2 엑셀 자동 생성
시뮬레이션 종료 시 자동으로 엑셀이 생성됩니다.
- 저장 경로: `source/extensions/morph.tbs_control_1/data/sim_logs/`
- 파일명 예: `sim_logs_YYYYmmdd_HHMMSS.xlsx`
- 시트 구성:
  - `진행현황`
  - `이력로그`
  - `애니메이션실행이력`

## 5. 애니메이션 JSON을 만드는 방법(시퀀스 편집기)

### 5.1 기본 원리
JSON은 “스텝 리스트”이며, 각 스텝은 동작 타입(MOVE/ROTATE/DELAY/USD_TIMELINE)과 파라미터(대상 prim, 시간, 이동량 등)를 가집니다. 실행 엔진은 스텝을 순서대로 실행해 애니메이션을 구성합니다.

### 5.2 추천 작업 순서
1) 시퀀스 편집기에서 대상 prim(또는 포트/LOT 오브젝트)을 선택/입력
2) MOVE/ROTATE 스텝으로 이동/회전 경로 정의
3) 필요하면 DELAY로 타이밍 조정
4) “현재 위치부터 시작”이 필요하면 체크 후 JSON 생성(스냅샷 저장)
5) JSON을 `data/sim_sequences/`에 저장
6) `event_animation_rules.json`에 이벤트 조건과 JSON 경로 매핑 추가

---

## 6. 규칙 기반 매핑(이벤트 → JSON)

### 6.1 두 가지 매핑 파일
- 권장(상태 기반): `config/event_animation_rules.json`
  - 이벤트/포트/from/to/포트점유 상태에 따라 다른 JSON을 선택 가능
- fallback(단순): `config/event_animation_map.json`

### 6.2 왜 rules가 필요한가
같은 이벤트라도 “현재 어떤 포트가 차 있는지/비었는지”에 따라, 다른 장면을 실행해야 할 수 있기 때문입니다.

---

## 7. 공정확인(게이트) 창에서 무엇을 확인하나

공정확인 창은 이벤트가 발생할 때 아래 순서로 정보를 보여주도록 구성되어 있습니다.

1) 발생 이벤트명
2) 이벤트 동작 설명
3) 연계 애니메이션 파일명 + 존재 여부 + 빈 파일 여부(또는 애니 불필요 안내)
4) 타이머(다음 생성/다음 회수티켓까지)
5) XML 원문

이 창을 통해 “왜 특정 애니메이션이 실행되는지/실행되지 않는지”를 이벤트/규칙/파일 상태 근거로 설명할 수 있습니다.

## 8. 보고서에 넣으면 좋은 캡처/영상 체크리스트

아래 항목은 “보고서/사내 공유”에서 이해도를 크게 올립니다.

### 8.1 핵심 화면 캡처(정적 이미지)
- **[캡처 A] 제어창 전체**: 진행현황/이력로그/애니메이션실행이력/포트상태가 한 화면에 보이게
- **[캡처 B] 공정확인 창**: 이벤트명, action_desc, 매핑된 JSON 파일명(존재/EMPTY 표시), TIMER, XML이 보이게
- **[캡처 C] 포트상태 패널**: BP/EP가 채워지고 비는 타이밍이 보이게 (특히 BP→EP 완료 순간)
- **[캡처 D] 시퀀스 편집기**: MOVE/ROTATE/DELAY 구성 예시 + “현재 위치부터 시작” 체크 구간
- **[캡처 E] rules.json**: 특정 이벤트 조건(예: from/to + ports_occupancy)로 JSON을 고르는 예시
- **[캡처 F] 엑셀 결과**: 3개 시트(진행현황/이력로그/애니메이션실행이력) 각각 1장씩

### 8.2 영상(짧은 클립 권장)
- **[영상 1] 30~60초 데모**: 시뮬 시작 → 이벤트 발생 → 애니 실행 → 포트상태 변화 → 로그 누적
- **[영상 2] 게이트 모드**: “각 공정 확인” 체크 후 이벤트마다 창이 뜨고 확인하면 다음 단계로 넘어가는 흐름
- **[영상 3] 시퀀스 제작→적용**: 시퀀스 편집기에서 JSON 저장 → rules에 연결 → 시뮬에서 자동 실행되는 장면

### 8.3 보고서 문장에 넣기 좋은 “설명 포인트”
- “이벤트는 공정의 사실을 기록하고, 애니메이션은 그 사실을 3D로 보여준다.”
- “rules 기반 매핑으로, 장면 변경은 코드 수정 없이 JSON/설정 중심으로 대응 가능하다.”
- “엑셀 산출물로 재현/검증 결과를 증적으로 남긴다.”
