# TBS 시뮬레이션 스토리 가이드

이 문서는 현재 `morph.tbs_control_1` 코드(`simulation_engine.py`)에 구현된 로직을 기준으로,
공정 흐름을 "책처럼" 읽을 수 있게 설명한 안내서입니다.

---

## 1) 등장인물과 무대

- **OHT**: LOT을 외부에서 받아 `BP1`에 내려놓고, 공정이 끝난 LOT을 `EP`에서 회수합니다.
- **BP1**: 입력 게이트 같은 포트. OHT가 먼저 LOT을 내려놓는 자리입니다.
- **BP2/BP3/BP4**: 버퍼 포트. 공정 전 대기 구역입니다.
- **EP1/EP2(/EP3)**: 실제 공정 포트. 여기서 `metro_time`만큼 공정을 수행합니다.
- **LOT**: `LOT_001` 같은 작업 단위입니다. 각 LOT은 `metro_time`을 가집니다.

핵심: 이 시뮬레이션에서 **OHT는 입력/회수 담당**,
`BP1->BP`, `BP->EP` 이동은 **별도 이송 동작(로봇팔 역할)**로 처리됩니다.

---

## 2) 시뮬레이션이 시작되면

시뮬레이션은 아래 순서로 열립니다.

1. 포트 구성 확정 (`EP` 2개 또는 3개)
2. 모든 포트를 기본 `READY_TO_LOAD / EMPTY`로 초기화
3. UI에서 체크한 초기 적재 포트가 있으면, 시작 시점에 해당 포트를 `FULL`로 채움
4. 남은 LOT은 입력 큐(`_oht_input_queue`)에 대기
5. 동시에 두 흐름이 돌기 시작
   - **입력 루프**: OHT가 LOT을 `BP1`으로 계속 투입
   - **공정 루프**: 버퍼 LOT을 EP로 보내 공정/회수

---

## 3) 한 LOT의 전체 여정 (기본 시나리오)

아래는 LOT 하나가 지나가는 전형적인 경로입니다.

### 장면 A: OHT가 BP1에 LOT을 내려놓는다

- 조건: `BP1`이 비어 있고, `BP2~4` 중 하나 이상 비어 있어야 입력 가능
- 동작: `OHT -> BP1` 이동
- 시간: `oht_to_bp1_min ~ max` 랜덤
- 이벤트/상태:
  - MOVE 계열 이벤트 발생
  - 도착 시 `BP1`이 `ARRIVED / FULL`

### 장면 B: BP1에서 버퍼(BP2/3/4)로 밀어 넣는다

- 조건: `BP2~4` 중 빈 포트 필요
- 선택 규칙: **가장 오래 비어 있던 버퍼 포트** 우선
- 동작: `BP1 -> BPx`
- 시간: `bp1_to_bp_min ~ max` 랜덤
- 상태 변화:
  - `BPx`는 `ARRIVED / FULL`
  - `BP1`은 `READY_TO_LOAD / EMPTY`

### 장면 C: 버퍼에서 공정포트(EP)로 보낸다

- 조건:
  - 비어 있는 EP가 있음
  - 버퍼에 LOT이 있음
- 선택 규칙:
  - EP: 비어 있는 포트 중 첫 번째
  - 버퍼: **가장 오래 기다린 LOT**
- 동작: `BPx -> EPy`
- 시간: `bp_to_ep_min ~ max` 랜덤
- 상태 변화:
  - `EPy`는 `ARRIVED / FULL`
  - `BPx`는 `READY_TO_LOAD / EMPTY`

### 장면 D: EP에서 공정 수행

- 동작: `process_ep_lot(ep_port, lot)`
- 시간: 해당 LOT의 `metro_time`
- 로그: 공정 시작/진행률/예상 완료

### 장면 E: 공정 완료 후 OHT가 회수

- 동작: `EPy -> OHT`
- 시간: `ep_to_oht_min ~ max` 랜덤
- 상태 변화:
  - `EPy`는 `READY_TO_LOAD / EMPTY`
  - LOT은 `completed_lots`에 추가

---

## 4) 여러 LOT이 있을 때의 동시 진행 감각

현실감은 "동시에 여러 줄기가 흐른다"에서 나옵니다.

- OHT는 다음 LOT을 BP1로 들여오는 중
- 다른 LOT은 버퍼에서 EP로 이동 중
- 또 다른 LOT은 EP에서 공정 중
- 공정이 끝난 LOT은 OHT 회수 대기/회수 중

그래서 로그를 보면 한 줄기만 직렬로 흐르지 않고,
시간축에서 여러 상태가 겹치며 진행됩니다.

---

## 5) "이럴 때는 이렇게" 상황별 예시

### 예시 1: EP가 모두 차 있으면?

- 버퍼 LOT은 EP로 못 들어갑니다.
- 로그에는 `WAIT`/`HEARTBEAT`가 찍히며 대기 상태를 보여줍니다.
- EP 중 하나가 비는 순간, 가장 오래 기다린 버퍼 LOT부터 진입합니다.

### 예시 2: BP2/3/4가 꽉 차 있으면?

- OHT가 BP1로 새 LOT을 넣는 입력은 잠시 멈춥니다.
- 버퍼가 비워지면 입력 루프가 자동 재개됩니다.

### 예시 3: 시작부터 특정 포트에 LOT이 있는 경우

- UI 초기 체크박스로 `BP3`, `EP1` 등을 체크하면
- 시작 시점에 해당 포트가 `FULL` 상태로 출발합니다.
- 그 상태를 포함해 나머지 큐 LOT이 이어서 투입/처리됩니다.

### 예시 4: EP를 3개 장비로 설정한 경우

- `EP1`, `EP2`, `EP3`를 공정 포트로 사용합니다.
- 버퍼 LOT이 더 빠르게 분산 처리될 수 있습니다.

---

## 6) 로그를 읽는 법

### 실시간 진행 로그

- `[INPUT]`: OHT 입력 관련
- `[MOVE]`, `[BP1->BUFFER]`: 포트 간 이송
- `[PROCESS]`: EP 공정 시작/진행
- `[READY_TO_UNLOAD]`, `[COMPLETE]`: 회수 준비/완료
- `[HEARTBEAT]`: 전체 진행 스냅샷
- `[PROGRESS]`: 긴 단계의 진행률

### 이벤트/애니메이션 훅 로그

- `[ANIM HOOK]`: 이벤트 발생 시점
- 포트 타입이 함께 표시됨:
  - `버퍼포트(BP)`, `공정포트(EP)`, `이송장치(OHT)`
- 향후 실제 애니메이션 분기는 `ANIM PLAN` 분기에서 연결

### 종료 요약 로그

- `[SUMMARY]`: 시뮬레이션 전체 종료 시각
- `[SUMMARY LOT]`: LOT별 단계 소요시간 + 실제 경로
  - 예: `BP1->BP3`, `BP3->EP2`, `EP2->OHT`

---

## 7) 한눈에 보는 전체 흐름

`OHT 입력 큐 -> BP1 -> BP2/3/4 버퍼 -> EP1/2(/3) 공정 -> OHT 회수 -> 완료`

이 사이클이 LOT마다 반복되고, 포트 여유/대기 상태에 따라 동시 진행됩니다.

---

## 8) 현재 로직 기준 확장 포인트

- 이벤트별 실제 애니메이션 함수 연결 (`handle_sim_event_for_animation` 분기)
- 포트명 ↔ 실제 prim path 매핑 테이블
- 요약 로그 JSON/CSV 저장
- 공정 정책 옵션 추가(우선순위, 장비별 스케줄링 룰)
